\section{Introduction}

The package {\ttfamily luacas} allows for symbolic computation within \LaTeX{}. For example:
\begin{CAS}
    vars('x','y')
    f = 3*x*y - x^2*y 
    fx = diff(f,x)
\end{CAS}
\inputminted[
firstline=7,
lastline=11,
breaklines]
{latex}
{docdemo.tex}
The above code will compute the partial derivative $f_x$ of the function $f$ defined by 
\[ f(x,y)=3xy-x^2y.\]
There are various methods for fetching and/or printing results from the CAS within your \LaTeX{} document:

\begin{codebox}
\inputminted[
firstline=14,
lastline=14,
breaklines]
{latex}
{docdemo.tex}
\tcblower
\[ \print*{fx} = \print{fx} \] 
\end{codebox}

\subsection{About}

The core CAS program is written purely in Lua and integrated into \LaTeX{} via Lua\LaTeX{}. Currently, most existing computer algebra systems such as Maple and Mathematica allow converting their stored expressions to \LaTeX{} code, but this still requires exporting code from \LaTeX{} to another program and importing it back, which can be tedious.

The target audience for this package are mathematics students, instructors, and professionals who would like some ability to perform basic symbolic computations within \LaTeX{} without the need for laborious and technical setup. But truly, this package was born out of a desire from the authors to learn more about symbolic computation. What you're looking at here is the proverbial ``carrot at the end of the stick'' to keep our learning moving forward.
                
Using a scripting language (like Lua) for the core CAS reduces performance dramatically, but the following considerations make it a good option for our intentions: 
                
\begin{itemize}
    \item Compiled languages that can communicate with \LaTeX{} in some way (such as C through Lua) require compiling the code on each machine before running, reducing portability.
    \item Our target usage would generally not involve computations that take longer than a second, such as factoring large primes or polynomials.
    \item Lua is a fast scripting language, especially when compared to Python, and is designed to be compact and portable.
    \item If C code could be used, we could tie into one of many open-source C symbolic calculators, but the point of this project is to learn the mathematics of symbolic computation. The barebones but friendly nature of Lua make it a fairly ideal language from a pedagogical point of view.
\end{itemize}

\subsection{Features}
                
Currently, {\ttfamily luacas} includes the following functionality:
                
\begin{itemize}
    \item Arbitrary-precision integer and rational arithmetic
    \item Number-theoretic algorithms for factoring integers and determining primality
    \item Constructors for arbitrary polynomial rings and integer mod rings, and arithmetic algorithms for both
    \item Factoring univariate polynomials over the rationals and over finite fields
    \item Polynomial decomposition and some multivariate functionality, such as pseudodivision
    \item Basic symbolic root finding
    \item Symbolic expression manipulations such as expansion, substitution, and simplification
    \item Symbolic differentiation and integration
\end{itemize}
                
The CAS is written using object-oriented Lua, so it is modular and would be easy to extend its functionality (which we hope to do in the future).

\section{Lua Documentation}

This section assumes some familiarity with the programming language Lua and with object-oriented programming. This section is geared towards those who are interested in the inner design, as opposed to the algorithms or mathematics, of the CAS. In particular, anyone who wishes to modify or expand the CAS for their own purposes should read this.

This computer algebra system is written in object-oriented Lua. Lua does not have built-in object-oriented features, but using metatables we can simulate much of the oo functionality of a language like Java. Every class in the CAS inherits from the {\ttfamily Expression} interface. Expressions are really trees, and may have any number of sub-expressions as children depending on the type of the expression.

Most operations that a user would want to apply to expressions, such as derivatives and integrals, are encapsulated into their own classes. These classes have the expressions that these operations are applied to as instance variables. Other kinds of operations on expressions are \emph{core methods}, which are methods in the objected-oriented sense, and can be called on expressions using the {\ttfamily :} syntax as is typical in Lua. The rule of thumb is that if there is standard mathematical notation for an operation or function represented by the CAS, it gets its own class, and if not, it is a core method. This way, every Lua expression corresponds to something that can be cleanly displayed in \LaTeX{}. Individual classes may also have methods unique to that class.

\subsection{Expression Types}

There are a sizeable number of expression types currently in the CAS. They are spread across the core, algebra, and calculus packages. If an expression has no possible sub-expressions, it is called \emph{atomic}. Otherwise, an expression is \emph{compound}. Atomic expressions and compound expressions are both subinterfaces to expressions, and all classes inherit from one of those interfaces.

A list of atomic expressions and their constructors are provided below.

\begin{itemize}
    \item {\ttfamily SymbolExpression(symbol)} takes a string and converts it into a symbol that represents an arbitrary value, such as $x$.
    
    \item {\ttfamily Integer(n)} takes a string or number and converts it into an arbitrary-precision integer, such as $-12$.
    
    \item {\ttfamily Rational(n, d)} takes an {\ttfamily Integer} numerator and denominator and constructs a rational number in reduced form, such as $\frac{3}{8}$. It may also take polynomial numerators and denominators, in which case it returns an element in the field of fractions over that polynomial ring, such as $\frac{x^2+3}{x-1}$. The {\ttfamily /} operator on two integers or polynomials can also be used to create a rational.
    
    \item {\ttfamily IntegerModN(i, n)} takes an {\ttfamily Integer} number and an {\ttfamily Integer} modulus and creates an element in the ring $\mathbf{Z}_n$, such as $4\mod 17$.
    
    \item {\ttfamily PolynomialRing(coefficients, symbol)} takes a list of coefficients in some ring and a symbol and produces a polynomial with those coefficients in a ring adjoined to that symbol, such as $5x^2-\frac{3}{2}x+\frac{1}{7}$.
\end{itemize}

Integers, rationals (not rational expressions), and integers mod an integer are \emph{constant expressions}, and inherit from the {\ttfamily ConstantExpression} interface. Those three expressions as well as polynomials and rational expressions inherit from the {\ttfamily Ring} interface.

A list of compound expressions and their constructors are provided below.

\begin{itemize}
    \item {\ttfamily BinaryOperation(operation, expressions)} takes a function and a list of expressions and creates an expression with that function as its operation, such as $3+2^{x-1}$. Associative binary operations may have any number of subexpressions, while non-associative binary operations must have exactly two subexpressions. The currently implemented binary operations are the ring operations $+$, $-$, $\times$, $\wedge$, $/$, $\%$, and $//$.
    
    \item {\ttfamily FunctionExpression(name, expressions)} takes a string name and a list of expressions and creates a generic function with its subexpressions as arguments, such as $f(3, y^2)$. 
    
    \item {\ttfamily AbsExpression(expression)} takes an expression and returns the absolute value of that expression, such as $|z|$.
    
    \item {\ttfamily SqrtExpression(expression, root)} takes two expressions and returns a root expression such as $\sqrt{3x+1}$. If {\ttfamily root} is ommited, it defaults to 2.
    
    \item {\ttfamily Logarithm(base, expression)} takes two expressions and returns a log expression such as $\log_2(x)$.
    
    \item {\ttfamily TrigExpression(name, expression)} takes a string name and expression and returns the appropriate trigonometric function applied to {\ttfamily expression}, such as $\sin(x)$. {\ttfamily TrigExpression} is a subclass of {\ttfamily FunctionExpression}. Valid trigonometric functions are sin, cos, tan, csc, sec, cot, arcsin, arccos, arctan, arccsc, arcsec, and arccot.
    
    \item {\ttfamily RootExpression(expression)} takes an expression and returns an expression representing the roots of the equation {\ttfamily expression} $= 0$, such as RootOf$(x^5+x-1)$.
    
    \item {\ttfamily DerivativeExpression(expression, symbol)} takes an expression and symbol and returns an expression for the single-variable derrivative of that expression with respect to the given symbol, such as $\frac{\mathrm{d}}{\mathrm{d}x}(x^y)$.
    
    \item {\ttfamily DiffExpression(expression, symbols)} takes an expression and list of symbols and returns an expression for the higher-order partial derivative of that expression with respect to each symbol, such as $\frac{\partial}{\partial x \partial y}x^y$.
    
    \item {\ttfamily IntExpression(expression, symbol)} takes an expression and symbol and returns an expression for the indefinite integral of the expression with respect to the given symbol, such as $\int \cos(x) \hspace{2pt} \mathrm{d}x$.
    
    \item {\ttfamily IntExpression(expression, symbol, lower, upper)} takes an expression and symbol and returns an expression for the definite integral of the expression with respect to the given symbol and bounds, such as $\int_{-2}^2 \cos(x) \hspace{2pt} \mathrm{d}x$.
\end{itemize}


\subsection{Core Methods}

All of the following core methods can be applied to any expression, and are defined in the {\ttfamily expression.lua} interface.


\begin{itemize}
    \item {\ttfamily evaluate()} applies an expression that represents an operation on expressions to its subexpressions. For instance, evaluating a {\ttfamily DerrivativeExpression} applies the derivative operator with respect to the {\ttfamily symbol} field to its {\ttfamily expression} field. Evaluating a {\ttfamily BinaryOperation} with its {\ttfamily operation} field set to $+$ returns the sum of the numbers in the {\ttfamily expressions} field, if all of the expressions are numbers. If the expression does not represent an operation or is unable to be evaluated, calling {\ttfamily evaluate()} on an expression returns itself.
    
    \item {\ttfamily autosimplify()} performs fast simplification techniques on an expression. All input from the \LaTeX{} end and everything output to the \LaTeX{} end is autosimplified. Generally, one should call {\ttfamily autosimplify()} on expressions before applying other core methods to them. The {\ttfamily autosimplify()} method also calls the {\ttfamily evaluate()} method.
    
    \item {\ttfamily simplify()} performs full simplification of an expression. This may involve polynomial factorization, which is slow, so this is seperate from autosimplification and not called on the \LaTeX{} end unless the user specifically directs the CAS to simplify. The {\ttfamily simplify()} method also calls the {\ttfamily autosimplify()} method.
    
    \item {\ttfamily subexpressions()} returns a list of all subexpressions of an expression. This gives a unified interface to the fields of subexpressions, which have different names across classes.
    
    \item {\ttfamily size()} returns the number of nodes of the tree that constitutes an expression, roughly the total number of expression objects that make up the expression.
    
    \item {\ttfamily setsubexpressions(subexpressions)} creates a copy of an expression with the list of subexpressions as its new subexpressions. This can reduce code duplication in other methods.
    
    \item {\ttfamily freeof(symbol)} determines whether or not an expression contains a particular \\ {\ttfamily SymbolExpression} somewhere in the tree.
    
    \item {\ttfamily substitute(map)} takes a table mapping expressions to other expressions and recursively maps each instance of an expression with its corresponding table expression.
    
    \item {\ttfamily expand()} expands an expression, turning sums of products into products of sums.
    
    \item {\ttfamily factor()} factors an expression, turning products of sums into sums of products.
    
    \item {\ttfamily isatomic()} determines whether an expression is atomic.
    
    \item {\ttfamily isconstant()} determines whether an expression is atomic and contains no variables.
    
    \item {\ttfamily isrealconstant()} determines whether an expression is a real number in the mathematical sense, such as $2$, $\sqrt{5}$, or $\sin(3)$.
    
    \item {\ttfamily iscomplexconstant()} determines whether an expression is a complex number in the mathematical sense, such as $3 + \sqrt{2}i$.
    
    \item {\ttfamily topolynomial()} attempts to convert a compound expression to a monovariate polynomial with type {\ttfamily PolynomialRing}.
    
    \item {\ttfamily tolatex()} converts an expression to string \LaTeX{} code.
\end{itemize}

The number of core methods should generally be kept small, since every new type of expression must implement all of these methods. The exception to this, of course, is core methods that call other core methods that provide a unified interface to expressions. For instance, {\ttfamily size()} calls {\ttfamily subexpressions()}, so it only needs to be implemented in the expression interface.

All expressions should also implement the {\ttfamily \_\_tostring} and {\ttfamily \_\_eq} metamethods. Unlike regular methods, metamethods cannot be inherited using Lua, thus every expression object created by a constructor must assign a metatable to that object.

\begin{itemize}
    \item {\ttfamily \_\_tostring} provides a human-readable version of an expression for printing within Lua. Used for debugging purposes.
    
    \item {\ttfamily \_\_eq} determines whether an expression is structurally identical to another expression.
\end{itemize}

\subsection{Ring Methods}

For those that wish to expand the algebraic structures the CAS is equipped with, the algebra packages come with interfaces for basic operations on rings and conversions between rings. This subsection assumes some familiarity with algebraic structures such as rings, Euclidean domains, and fields.

Each ring, domain, and field operation also has a corresponding metamethod to use as a shorthand. These metamethods are generic, and handle converting input arguments. The regular methods assume all arguments are part of the same ring.

The ring operations and corresponding metamethods are:

\begin{itemize}
    \item {\ttfamily a:add(b)} ({\ttfamily a + b}) adds {\ttfamily a} and {\ttfamily b}.
    \item {\ttfamily a:sub(b)} ({\ttfamily a - b}) subtracts {\ttfamily b} from {\ttfamily a}.
    \item {\ttfamily a:neg()} ({\ttfamily -a}) returns the additive inverse of {\ttfamily a}.
    \item {\ttfamily a:mul(b)} ({\ttfamily a * b}) multiplies {\ttfamily a} and {\ttfamily b}.
    \item {\ttfamily a:pow(n)} ({\ttfamily a \wedge{} n}) raises {\ttfamily a} to the power of an integer {\ttfamily n}.
    \item {\ttfamily a:eq(b)} ({\ttfamily a == b}) returns whether {\ttfamily a} and b are the same element of a ring.
    \item {\ttfamily a:lt(b)} ({\ttfamily a < b}) returns whether {\ttfamily a} is less than {\ttfamily b} under some total order on ring elements. Implementation of this method is not necessary if no natural total order exists on the given ring. 
    \item {\ttfamily a:le(b)} ({\ttfamily a <= b}) returns whether {\ttfamily a} is less than or equal to {\ttfamily b} under some total order on ring elements. Implementation of this method is not necessary if no natural total order exists on the given ring.
    \item {\ttfamily a:zero()} returns the additive  identity of a ring.
    \item {\ttfamily a:one()} returns the multiplicative identity of a ring.
\end{itemize}

Subtraction and exponentiation have default implementations in the ring interface, but these can be overridden. Exponentiation is implemented as repeated multiplication, so overriding this method with a more efficient approach is recommended.

Euclidean domains are subinterfaces to rings, and require the following methods in addition to ring methods:

\begin{itemize}
    \item {\ttfamily a:divremainder(b)} ({\ttfamily a // b} and {\ttfamily a \% b}) performs division with remainder on {\ttfamily a} and {\ttfamily b}, and returns both the quotient and the remainder. Using {\ttfamily //} returns only the quotient, while using {\ttfamily \%} returns only the remainder.
\end{itemize}

Fields are subinterfaces to Euclidean domains, and require the following methods in addition to Euclidean domains methods:

\begin{itemize}
    \item {\ttfamily a:div(b)} ({\ttfamily a / b}) divides {\ttfamily a} by {\ttfamily b}.
    \item {\ttfamily a:inv()} returns the multiplicative inverse of {\ttfamily a}.
\end{itemize}

Division has a default implementation in the field interface, but this can be overridden.

It is worth mentioning how binary operations on different rings are implemented. The code makes use of a {\ttfamily RingIdentifier} object that acts as a sort of class without any methods. This is used because a single class can instantiate objects in multiple rings, such as polynomial rings, which can be constructed over any ring and any variable, such as $\mathbf{Z}[x]$ or $\mathbf{Q}[y]$.

The {\ttfamily Ring} class also requires the following methods for dealing with {\ttfamily RingIdentifier}s:

\begin{itemize}
    \item {\ttfamily Ring.makering()} is a static method that returns the default {\ttfamily RingIdentifier} for the {\ttfamily Ring}.
    
    \item {\ttfamily a:getring()} returns the particular {\ttfamily RingIdentifier} the element {\ttfamily a} is part of.  For classes with only one possible ring, such as the {\ttfamily Integer} class, this does the same thing as {\ttfamily makering}.
    
    \item {\ttfamily a:inring(ring)} takes a {\ttfamily RingIdentifier} ring and converts {\ttfamily a} to an element of that ring, if such a conversion is feasible. 
\end{itemize}

Finally, the static method {\ttfamily Ring.resultantring(ring1, ring2)} calculates the smallest {\ttfamily RingIdentifier} that contains {\ttfamily ring1} and {\ttfamily ring2} as subrings. The full process for performing a binary operation, say, multiplication, on two elements of possible different rings looks like:

\begin{minted}{lua}
    local aring, bring = a:getring(), b:getring()
    local oring = Ring.resultantring(aring, bring)
    if not oring then
        error("Attempted to perform multiplication on two elements of incompatable rings")
    end
    return a:inring(oring):mul(b:inring(oring))
\end{minted}

One should be careful about what kind of conversions are allowed. Converting an integer to a $1\times1$ matrix likely makes sense, but converting a polynomial to matrix does not.

\subsection{Constructor Shorthands \& Basic Parsing}

Algebraic computations on the Lua end typically involve constructing the desired expression and applying one or more operations to the expression. If a non-core operation represented by a class is applied, then this operation will be {\ttfamily autosimplify()}.

Naively, one could nest class constructors to build an expression. For instance, the Lua code for evaluating the integral \[ \int_0^6 \sin(x) + 2x^2 + 3 \hspace{2pt}\mathrm{d}x\] and storing the result in a variable {\ttfamily r} is as follows:

\begin{minted}{lua}
r = IntegralExpression(BinaryOperation(BinaryOperation.ADD,
            {TrigExpression("sin", SymbolExpression("x")),
            BinaryOperation(BinaryOperation.MUL,
                {Integer(2),
                BinaryOperation(BinaryOperation.POW,
                {SymbolExpression("x"), Integer(2)})}),
            Integer(3)}),
        SymbolExpression("x"),
        Integer(0),
        Integer(6)):autosimplify()
\end{minted}

This is more than unwieldy, so a number of constructs are added to make using and developing the CAS easier. Metatables for binary operations allow natural symbols to act as constructors that use Lua's build-in operator precedence:

\begin{minted}{lua}
r = IntegralExpression(TrigExpression("sin", SymbolExpression("x")) +
        Integer(2) * SymbolExpression("x") ^ Integer(2) +
        Integer(3),
        SymbolExpression("x"),
        Integer(0),
        Integer(6)):autosimplify()
\end{minted}

Several functions are added as shorthand for constructors that mirror familiar syntax. These function names are in all caps to reduce the risk of collisions.

\begin{minted}{lua}
r = INT(SIN(SymbolExpression("x")) + Integer(2) * SymbolExpression("x") ^ Integer(2) 
    + Integer(3),
        SymbolExpression("x"),
        Integer(0),
        Integer(6)):autosimplify()
\end{minted}

This is still fairly unwieldy, mostly because every string and number need to be wrapped in a constructor to convert them to expressions. Unfortunately, this syntax is also the most awkward to avoid in Lua, and the inner workings of the code just uses the longer constructors.

For \LaTeX{} users, or for Lua users who are looking for a simpler paradigm, a parser was implemented that takes the code input as a string and wraps numbers in the {\ttfamily Integer} constructor, then evaluates the code using Lua's {\ttfamily load()} function. The {\ttfamily vars()} command also allows variables to be declared as symbol expressions with the same name, at the cost of being able to use these symbols to reference other objects. Using the parser, the code now looks like:

\begin{minted}{lua}
CASparse([[
    vars("x")
    r = int(sin(x) + 2*x^2 + 3, x, 0, 6)
]])

\end{minted}

The CAS \LaTeX{} environment is implemented by wrapping the text inside the environment with the {\ttfamily CASparse} command.
